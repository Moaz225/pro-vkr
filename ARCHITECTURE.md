## 2. Описание архитектуры

### 2.1. Архитектурный подход (паттерн)

Проект реализован как **слоистое (layered) монолитное web-приложение** с «тонким» клиентом:

- **Представление (Presentation layer)**: статические HTML-страницы и клиентский JavaScript для гостевого интерфейса и панелей персонала/менеджера (`index.html`, `script.js`, `staff-orders.html`, `manager.html`).
- **Прикладная логика (Application layer)**: единый backend на Express, содержащий REST API, бизнес-правила по заказам/бронированиям/платежам, а также middleware безопасности (`server/server.js`).
- **Доступ к данным (Data access layer)**: Prisma ORM, описывающая модели предметной области и обеспечивающая операции с базой данных (`prisma/schema.prisma`, использование Prisma Client в `server/server.js`).

Архитектура **не является микросервисной**: все ключевые функции (авторизация, оформление заказа, бронирование, создание платежа, обработка вебхуков) сосредоточены в одном серверном приложении.

### 2.2. Компоненты системы

**Клиентские компоненты:**
- **Гостевой интерфейс**: меню, корзина, оформление заказа, ввод номера столика и комментария (`index.html`, логика — `script.js`).
- **Панель персонала**: мониторинг заказов/бронирований, изменение статусов заказов (`staff-orders.html`).
- **Панель менеджера**: агрегирование показателей, график продаж (`GET /api/analytics/sales`), CRUD товаров (архив вместо удаления), стоп-лист, просмотр сущностей, управление статусами бронирований (`manager.html`).

**Серверные компоненты (Express):**
- **REST API** для заказов, бронирований, авторизации, CSRF и health-check (реализация маршрутов — `server/server.js`).
- **Платёжный модуль YooKassa**: создание платежа, формирование `return_url`, обработка webhook-уведомлений и подтверждение статусов через повторную проверку (реализация — `server/server.js`, конфигурация — `.env.example`).
- **Подсистема безопасности**: Helmet, CORS allowlist, rate limiting, CSRF, cookie-сессии (реализация — `server/server.js`).

**Компоненты данных:**
- **PostgreSQL** как основное хранилище.
- **Prisma schema** с моделями `User`, `Order`, `OrderItem`, `Reservation`, `Payment`, `WebhookEvent` (`prisma/schema.prisma`).

### 2.3. Протоколы и форматы обмена

- **Клиент ↔ Сервер**: HTTP/HTTPS, обмен данными в формате **JSON** (используется `fetch` в `script.js` и встроенных скриптах HTML).
- **Сервер ↔ База данных**: Prisma поверх PostgreSQL (конфигурация подключения через `DATABASE_URL` в `.env.example`).
- **Сервер ↔ YooKassa**:
  - исходящие запросы по HTTPS к API YooKassa для создания и проверки платежей (`server/server.js`);
  - входящие HTTPS webhook-уведомления на endpoint сервера (`server/server.js`).

## 3. Инфраструктура

### 3.1. Хостинг и развёртывание

В текущем репозитории отсутствуют артефакты контейнеризации/CI (например, `Dockerfile`, `.github/workflows/*`). Следовательно, базовая модель развертывания предполагает:

- запуск Node.js-приложения как процесса (service) на целевой платформе;
- обратный прокси и TLS-терминацию на уровне инфраструктуры (типовая практика для production);
- настройку переменных окружения по шаблону `.env.example`.

Для локального тестирования webhooks возможна публикация локального сервера в интернет через туннелирование (упоминается переменная `PUBLIC_BASE_URL` в `.env.example`, используется в `server/server.js` для построения `return_url`).

### 3.2. База данных

- **PostgreSQL** используется как единое централизованное хранилище.
- Схема данных описана в `prisma/schema.prisma`, миграции управляются Prisma.
- Сессии аутентификации хранятся в PostgreSQL через `connect-pg-simple` (конфигурация store — `server/server.js`).

### 3.3. Кэширование

Специализированное кэширование (например, Redis) **не реализовано**. Все чтения выполняются напрямую из PostgreSQL через Prisma и REST API.

### 3.4. Очереди сообщений

Очереди сообщений (RabbitMQ/Kafka/SQS и др.) **не используются**. Надёжность событий оплаты достигается за счёт:

- webhook-уведомлений YooKassa;
- дедупликации событий на стороне сервера с помощью таблицы `WebhookEvent` и уникального `dedupeKey` (см. `prisma/schema.prisma`, обработка — `server/server.js`).

## 4. Меры безопасности

### 4.1. Аутентификация и управление сессиями

- Метод аутентификации: **cookie-based sessions** через `express-session` (см. `server/server.js`).
- Cookie сессии конфигурируется как **HttpOnly**, что снижает риск доступа из клиентского JavaScript при XSS.
- Серверное хранилище сессий: PostgreSQL (`connect-pg-simple` + `pg`), что повышает устойчивость в production по сравнению с MemoryStore (см. `server/server.js`).

### 4.2. Защита от CSRF

Поскольку аутентификация использует cookies, реализована защита от CSRF:

- выдача CSRF-токена: `GET /api/csrf` (`server/server.js`);
- передача токена клиентом в заголовке `X-CSRF-Token` для state-changing запросов (`script.js`, `staff-orders.html`, `manager.html`);
- обработка ошибок CSRF на сервере (middleware-обработчик в `server/server.js`).

### 4.3. Защита API (header hardening, CORS, rate limiting)

- **Helmet**: формирование набора защитных HTTP-заголовков (см. `server/server.js`).
- **CORS**: контроль источников запросов и включение `credentials: true` для поддержки cookie-сессий (см. `server/server.js`, параметр `CORS_ORIGIN` в `.env.example`).
- **Rate limiting**: ограничение частоты запросов на уровне middleware (см. `server/server.js`).

### 4.4. Защита данных и платежной информации

- Проект **не хранит данные банковских карт**. Платёжные операции выполняются через YooKassa; в БД сохраняются только технические идентификаторы/статусы платежей и служебные поля, необходимые для корректной логики приложения (`Payment` в `prisma/schema.prisma`, обработка — `server/server.js`).
- Для обеспечения целостности обработки оплаты используется:
  - повторная проверка состояния платежа на стороне сервера (defense-in-depth для входящих вебхуков);
  - идемпотентность webhook-обработки через таблицу `WebhookEvent` и уникальный `dedupeKey` (`prisma/schema.prisma`, `server/server.js`).
